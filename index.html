Android 单元测试

单元测试是开发人员需要自己完成的一个测试，可能有时候我们会觉得写测试会特别费时间，毕竟代码写下去并运行后我们还是可以保证它的正确性的。但是当项目被频繁的修改的时候就会有很多莫名奇妙的 bug 冒出来了，定位一个 bug 至少都需要经历重新编译一次程序然后再安装到手机上并重现 bug 的场景的流程，因此适当的写一些单元测试还是可以节省调试时间的。

什么是单元测试

  • 跟普通的代码相比，它是验证代码正确性的代码。
  • 它是由开发人员自己编写的测试代码。
  • 单元测试可便于后期的重构和维护
  • 可优化设计，单元测试能使开发人员的代码便于测试的方向优化，解除软件中的耦合。
  • 单元测试也是一种特殊的文档，它可展示函数或则类如何使用。

Junit 简介

Junit 是基于 java 语言的单元测试框架，已默认配置在安卓项目中。

  1. 使用时的注意事项
  • 测试的方法应该是 public 的
  • 可选用 setUp 进行初始化工作， 使用 tearDown 进行收尾工作。
  • 每个测试方法没有关联，执行的顺序也不一定是代码中的顺序，因此测试方法之间不要存在有依赖性。

  • 是否为同一个对象 assertSame

  • 失败提示

Android 中的 MVC

简述

现在市面上很多成功的 Android 工程框架都来自于早期的 MVC 模型因此了解 MVC 还是很有必要的。 MVC 即 Model-View-Controller 中文名为模型视图控制器，是早期二十世纪八十年代为桌面应用建立起来的一个框架，现在反而在 web 引用中大放光彩。就拿 java 后台开发来说也衍生出了 SSH (Struts + Spring + Hibernate) 、SSM 、 SpringMVC 等优秀的框架。它们完美的将数据库数据、控制逻辑、以及界面的显示区分开来。但是在 Android 开发中很多时候都分不清这三者的界限导致 Activity 上的代码过于臃肿，修改起来费时费力。

相关名词

  • 视图逻辑
    比如界面想显示年龄但在后台获取到是数据是出生年月日，则视图逻辑就要对日期做相应的转换并显示出年龄。
  • 业务逻辑
    对数据的一系列操纵，如查询数据列表、增加数据、修改数据、删除数据。

三层的分工

  • Model: 即数据层, 负责处理业务逻辑, 监听网络与数据库接口.
  • View: 即界面 ( UI ) 层, 显示来源于 Model 的数据.
  • Contoller: 即逻辑层, 传递用户的交互和更新 Model 的数据.

根据 MVC 架构, 视图层和逻辑层都会依赖于数据层, 视图层显示模型层的数据, 逻辑层更新数据层数据。数据层从项目中分离后, 独立于 UI , 允许测试. 更新数据层方式的不同, 把 MVC 架构分为被动 ( Passive ) 模式和主动 ( Active ) 模式.

被动模式

在被动模式中, 逻辑层是唯一操作数据层的类。基于用户的响应事件, 逻辑层通知数据层更新数据。在数据层更新后, 逻辑层通知视图层更新 UI , 视图层从数据层中获取数据。

主动模式

在主动模式中, 逻辑层不是唯一操作数据层的类, 数据层存在自更新机制。在更新数据时, 数据层层使用观察者 ( Observer ) 模式通知视图层和其他类. 视图层实现观察者的接口, 在数据层中, 注册成为观察者, 接收通知。

当数据层发生数据更新时, 告知全部观察者。视图层从数据层中更新数据。

在被动模式中, 视图层继承基类 BaseView , 逻辑层拥有 BaseView 的引用, 更新数据;
在主动模式中, 数据层使用观察者通知视图层, 更新数据。

优点

MVC 模式, 分离类的 UI 与业务职责, 增加可测试性与可扩展性. 数据层不引用任何 Android 类, 允许单元测试 ( Unit Test ) 。逻辑层含有视图层的引用, 不引用 Android 类, 允许单元测试. 视图层满足单一职责原则 ( SRP ) , 传递事件至逻辑层, 展示数据层数据, 不包含业务逻辑, 允许UI测试。

缺点

视图层既依赖于逻辑层又依赖于数据层. 在修改 UI 逻辑时, 也需要修改数据层, 降低架构的灵活性. 视图层与数据层的职责部分重叠, 过于耦合, 在处理 UI 逻辑时, 被动模式与主动模式都会产生若干问题.

Android ConstraintLayout

一、传统界面布局

在安卓开发中，界面编程是一个很重要的部分，而很多时候为了实现好看的 ui 界面会用到很多层的布局来实现，当一个应用的界面含有多个控件并且布局嵌套的层级大于三层时可能界面的渲染速度就会变慢，导致界面丢帧，这时界面加载起来就会很不流畅。极端情况下也会发生 ANR (Application Not Responding) 。因此一个优秀的界面应该尽量的去满足以下两点。

  • 布局嵌套的层级经可能的少
  • 布局中的控件应该尽可能的使用相对位置。
    使用相对位置可以在不用分辨率的设备中保持布局效果的一致性。

但是以上两点往往是冲突的，降低嵌套层级的方式是使用 RelativeLayout, 而基于相对位置使用的是 LinearLayout。当界面过于复杂时需要大量的 LinearLayout 保证控件的相对摆放，这样会导致嵌套层级过多，如果使用 RelativeLayout控件之间的关系及其冗余,这时的界面可读性差也难与修改。

二、ConstraintLayout布局

ConstraintLayout 即约束布局，在2016年的 Google I/O 大会上推出，可一定程度的替代掉 RelativeLayout 和 LinearLayout 接下来就开始了解一些简单的用法。

Android CoordinatorLayout

CoordinatorLayout 协调布局，可以理解为功能更强大的 FrameLayout 布局。它在普通情况下作用和 FrameLayout 基本一致，通常适用于两种使用方式：第一，最为界面最顶层的装饰布局；第二，作为包含一个或多个子视图的特定交互容器。
CoordinatorLayout 可以监听其所有子控件的各种事件，然后帮我们做出最为合理的响应。

在 FrameLayout 布局的右下角放了一个悬浮按钮

为其添加点击事件，当点击按钮时弹出一个 Snackbar 提示框。

可以看到如果使用的是普通的 FrameLayout 当父布局则提示框弹出时会有一小部分挡住悬浮按钮， 要是以前可以选择调整 layout_margin 来避免这类情况的发生，但现在有了更优雅的解决方法，只需将布局文件里的 FrameLayout 改成 android.support.design.widget.CoordinatorLayout 即可。

  • CoordinatorLayout 本身就是一个加强版的 FrameLayout，所以这种替换不会有任何副作用。
  • Snackbar 虽然不是 CoordinatorLayout 的子控件，但是因其在调用时在 make() 里传入了一个 View 对象，这个对象是在浮动按钮的监听事件里获取的，而浮动按钮又是 CoordinatorLayout 中的子控件，所以 Snackbar 也能被其监听到。

Hexo入门

使用

  • hexo init //初始化一个文件夹，它会为这个文件夹配置所有骨架
  • hexo g //即hexo generate，生成静态文件
  • hexo s //即hexo server，创建服务
  • hexo d //即hexo deploy,用于将本地文件发布到github上
  • hexo n //即hexo new,用于新建一篇文章

安装x

  • ⑴安装Nodejs：
    Node.js的官网下载地址是：https://nodejs.org/
    下载好之后，双击安装，一路Next即可

  • ⑵安装Git for windows：
    Git for windows 的下载地址是：https://git-for-windows.github.io/
    下载好后也同样是一路Next安装即可。

  • ⑶安装Hexo：
    1、 打开Git Bash，输入：npm install hexo-cli -g。
    2、 进入文件加D:/MyBlog，右键Git Bash Here，输入hexo init。输入npm install。输入hexo g。输入hexo s。
    3、浏览器输入地址localhost:4000查看。

  • ⑷部署：
    配置好后，输入hexo g。输入hexo d。
    如果有错则输入npm install hexo-deployer-git --save。
