Android 中的 MVC

简述

现在市面上很多成功的 Android 工程框架都来自于早期的 MVC 模型因此了解 MVC 还是很有必要的。 MVC 即 Model-View-Controller 中文名为模型视图控制器，是早期二十世纪八十年代为桌面应用建立起来的一个框架，现在反而在 web 引用中大放光彩。就拿 java 后台开发来说也衍生出了 SSH (Struts + Spring + Hibernate) 、SSM 、 SpringMVC 等优秀的框架。它们完美的将数据库数据、控制逻辑、以及界面的显示区分开来。但是在 Android 开发中很多时候都分不清这三者的界限导致 Activity 上的代码过于臃肿，修改起来费时费力。

相关名词

  • 视图逻辑
    比如界面想显示年龄但在后台获取到是数据是出生年月日，则视图逻辑就要对日期做相应的转换并显示出年龄。
  • 业务逻辑
    对数据的一系列操纵，如查询数据列表、增加数据、修改数据、删除数据。

三层的分工

  • Model: 即数据层, 负责处理业务逻辑, 监听网络与数据库接口.
  • View: 即界面 ( UI ) 层, 显示来源于 Model 的数据.
  • Contoller: 即逻辑层, 传递用户的交互和更新 Model 的数据.

根据 MVC 架构, 视图层和逻辑层都会依赖于数据层, 视图层显示模型层的数据, 逻辑层更新数据层数据。数据层从项目中分离后, 独立于 UI , 允许测试. 更新数据层方式的不同, 把 MVC 架构分为被动 ( Passive ) 模式和主动 ( Active ) 模式.

被动模式

在被动模式中, 逻辑层是唯一操作数据层的类。基于用户的响应事件, 逻辑层通知数据层更新数据。在数据层更新后, 逻辑层通知视图层更新 UI , 视图层从数据层中获取数据。

主动模式

在主动模式中, 逻辑层不是唯一操作数据层的类, 数据层存在自更新机制。在更新数据时, 数据层层使用观察者 ( Observer ) 模式通知视图层和其他类. 视图层实现观察者的接口, 在数据层中, 注册成为观察者, 接收通知。

当数据层发生数据更新时, 告知全部观察者。视图层从数据层中更新数据。

在被动模式中, 视图层继承基类 BaseView , 逻辑层拥有 BaseView 的引用, 更新数据;
在主动模式中, 数据层使用观察者通知视图层, 更新数据。

优点

MVC 模式, 分离类的 UI 与业务职责, 增加可测试性与可扩展性. 数据层不引用任何 Android 类, 允许单元测试 ( Unit Test ) 。逻辑层含有视图层的引用, 不引用 Android 类, 允许单元测试. 视图层满足单一职责原则 ( SRP ) , 传递事件至逻辑层, 展示数据层数据, 不包含业务逻辑, 允许UI测试。

缺点

视图层既依赖于逻辑层又依赖于数据层. 在修改 UI 逻辑时, 也需要修改数据层, 降低架构的灵活性. 视图层与数据层的职责部分重叠, 过于耦合, 在处理 UI 逻辑时, 被动模式与主动模式都会产生若干问题.
